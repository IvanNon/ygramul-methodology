# Ygramul Doc 2 — The 11-Lever Framework Deep

**Audience:** Internal — Claude (or human auditor) running Ygramul audits.
**Purpose:** The working reference for applying the 11 levers during audit execution. Each lever has the diagnostics that matter, the failure modes to scan for, the cross-lever interactions to flag, and the data sources required to evaluate it.
**Inherits from:** `ygramul-codex-architecture.md` (system reference) + `LEVERS.md` in the public methodology repo (canonical lever definitions).
**Replaces:** `ygramul-doc-2-7-lever-framework-deep.md` (deprecated — built on the outdated 7-lever model).
**Last updated:** May 4, 2026 — rebuilt from scratch for the 11-lever model.

---

## How to use this doc

This is the operational layer between the public `LEVERS.md` (which is methodology-facing, descriptive, suitable for senior practitioners reading the framework cold) and the audit deliverables that reach clients (which are evidence-led, prioritized, and tied to specific URLs).

**For Claude running an audit:** Read this doc once at session start. When applying a lever, jump to that lever's chapter. The chapter tells you what signals to evaluate, what data you need, what failure modes to scan for, and how to cross-reference the finding to other levers.

**For human auditors:** Same. The chapter structure is built so that you can land in any lever and have everything needed to evaluate it without bouncing between four other documents.

**Hierarchy of authority:** If `LEVERS.md` updates, `LEVERS.md` wins — patch this doc to match. If `ygramul-codex-architecture.md` and this doc disagree on system-level questions (agent inventory, RAG architecture, build phases), the architecture doc wins. This doc is downstream of both.

## Common structure for each lever chapter

Every lever chapter below has the same eight sections:

1. **What this lever covers** — one-paragraph scope statement
2. **Why it sits where it sits** — sequencing justification and dependencies
3. **What success looks like** — the steady-state target
4. **Diagnostic signals to evaluate** — the specific signals to check during an audit
5. **Data sources required** — what data needs to be in hand to evaluate this lever
6. **Common failure modes** — patterns observed in production
7. **Cross-lever interactions** — which other levers' issues compound with this one
8. **Default severity weights** — how findings in this lever get prioritized P0/P1/P2/P3 absent client-specific context

The eight-section structure is deliberate. It makes the lever chapters mechanically usable during audit execution.

## Severity framework (used across all levers)

Findings are prioritized P0–P3. The framework is documented in Doc 1 master prompt; reproduced here for reference.

| Priority | Definition | Example |
|---|---|---|
| **P0** | Critical — actively destroying value or blocking other work. Fix immediately. | `_rsc` phantom URL crawl trap consuming 71% of crawl budget. Canonical pointing to wrong URL on top revenue pages. |
| **P1** | High — meaningful traffic or rank impact. Fix in current sprint. | Missing schema on author pages where E-E-A-T signal is needed. Redirect chain >3 hops on commercial templates. |
| **P2** | Medium — measurable but not urgent. Fix in current quarter. | Inconsistent anchor text across internal links. Suboptimal heading hierarchy on a content cluster. |
| **P3** | Low — nice to have. Fix opportunistically. | Image alt text could be more descriptive on a long-tail content section. |

**Severity is contextual.** A P1 on one site is a P2 on another. The default weights below are starting points, not rules. Adjust based on traffic/revenue exposure, the client's commercial priorities, and the implementation cost.

## Lever 1 — Crawl Architecture

### What this lever covers

The site-level work that determines how a search engine's crawler navigates the site as a whole. What gets crawled, how often, and whether crawl budget is spent on URLs that matter or wasted on URLs that don't.

### Why it sits at position 1

Everything downstream depends on the crawler reaching the right URLs at the right cadence. If Lever 1 is broken, the most important page on the site might not be indexed. Optimization at any other lever assumes this one is solved. When Lever 1 is broken, work at Levers 2 through 10 is wasted budget.

### What success looks like

The crawler reaches every URL that should be indexed, on a cadence appropriate to that URL's update frequency, without wasting crawl budget on parameter variants, infinite spaces, or phantom URLs generated by JavaScript framework artifacts. Server logs show a crawl distribution that matches the site's actual content priority. The indexation rate — valid URLs that are actually indexed, divided by valid URLs that should be — is high and stable.

### Diagnostic signals to evaluate

- **robots.txt directives.** Are parameter handling rules current with the site's URL surface? When was the last update?
- **XML sitemap coverage vs. crawled URL coverage.** Segmented sitemaps by template/section reveal coverage gaps that single-sitemap audits miss.
- **Crawl frequency by URL pattern** (server log analysis). Does the distribution match content priority?
- **Phantom URL generation patterns.** JS framework prefetch artifacts (especially Next.js App Router `_rsc`), parameter pollution, faceted navigation generating infinite spaces.
- **Crawl path efficiency.** Click depth from homepage to commercially valuable URLs. Are they reachable in ≤3 clicks?
- **Indexation rate trend.** GSC Page Indexing report over the last 6–12 months. Is "Crawled — currently not indexed" growing?
- **Crawl budget allocation.** What share is going to commercial pages, navigational pages, and waste?

### Data sources required

- **Critical:** Server logs (Cloudflare, Vercel, Apache/Nginx). Without server logs, crawl analysis is inferred rather than measured.
- **Critical:** GSC Page Indexing report (full export, segmented by URL pattern).
- **Important:** Screaming Frog crawl with JS rendering enabled, configured to match Googlebot Mobile.
- **Important:** robots.txt current state + change history if available.
- **Useful:** Sitemap files (XML), preferably segmented.

If server logs are unavailable, declare the gap in the audit and note that crawl analysis is inference-based rather than measurement-based. Don't fabricate confidence you don't have.

### Common failure modes

- **robots.txt treated as a binary.** Auditors check whether it exists, not whether parameter handling rules have evolved with the URL surface. New framework features ship; crawl waste accumulates silently.
- **Sitemap segmentation missing.** A single 50,000-URL sitemap obscures coverage gaps. Segmented sitemaps surface them immediately.
- **Phantom URLs dismissed as minor.** Parameter pollution and prefetch artifacts can multiply the URL surface 100×. The cost is invisible in standard crawl tools and only appears in log analysis or in the GSC "Crawled — currently not indexed" bucket.
- **Server logs never analyzed.** Crawl behavior inferred from third-party tools rather than read directly from where the crawler actually visited.

### Cross-lever interactions

- **Lever 2 (Canonical Architecture):** Phantom URL discovery is often a Lever 2 issue manifesting as a Lever 1 symptom. The canonical tag is the primary mechanism that prevents indexation drift on phantom variants. **Sequencing:** fix canonicals first (Lever 2), then validate crawl recovery (Lever 1).
- **Lever 6 (Programmatic SEO):** Faceted programmatic surfaces multiply crawl waste if Lever 6 is implemented before Lever 1's parameter handling is solid.
- **Lever 11 (Auto-Research):** Crawl waste fixes show indexation recovery in 30 days — fast enough to feed back cleanly into the auto-research loop.

### Default severity weights

- Phantom URL surface >10× real URL count → **P0**
- Indexation rate <60% on commercial templates → **P0**
- robots.txt missing rules for known-noisy parameters → **P1**
- Sitemap unsegmented on a 50K+ URL site → **P2**
- Crawl path depth >4 from homepage on commercial URLs → **P2**

## Lever 2 — Canonical Architecture

### What this lever covers

The decisions about which URL patterns are authoritative for which content, and how those decisions are signaled to search engines. Both the strategic layer (what the canonical URL structure should be) and the implementation layer (canonical tags expressing that structure on each page).

### Why it sits at position 2

Canonical decisions are upstream of every other on-page signal. The canonical tag, the redirect strategy, the schema markup, the internal link target — all implement decisions that were made (or should have been made) at the canonical architecture level. If the architecture is wrong, downstream signals contradict each other.

This is also why Lever 2 sits before Lever 3 (Page Technical Signals). The per-page implementation in Lever 3 cannot precede the architecture decisions in Lever 2.

### What success looks like

Every page has exactly one canonical URL. That URL is the one that should rank. The canonical signal is consistent across canonical tags, internal links, sitemaps, hreflang annotations, and structured data. Locale variants resolve cleanly. Pagination canonicalizes correctly (typically self-canonical, not to page 1). Filter and sort URL variants either canonicalize to the parent category or are blocked from indexation entirely.

### Diagnostic signals to evaluate

- **Canonical tag presence and accuracy** across templates. Sample at least one URL per major template.
- **Locale prefix handling.** Look for `/en/en/`, `/ja/ja/`, double-prefix bugs. These are framework artifacts that hit a small share of URLs but often the most commercially valuable ones.
- **Pagination canonical strategy.** Self-canonical (preferred) vs. canonical to page 1 (deprecated by Google).
- **Filter and sort URL canonical handling.** Documented strategy or ad hoc per-template?
- **Cross-domain canonical signals** (where syndication or content sharing applies).
- **Canonical vs. internal link consistency.** The canonical points to URL X but internal links point to URL Y. The crawler treats this as a contradiction.
- **hreflang and canonical alignment.** hreflang annotations should reference canonical URLs, not their variants.

### Data sources required

- **Critical:** Screaming Frog crawl with canonical column exported.
- **Critical:** Sample inspection of 5–10 commercial URLs per template, manually checking canonical, hreflang, and internal link targets together.
- **Important:** GSC URL Inspection on suspect URLs to see Google's interpreted canonical.
- **Useful:** Sitemap URLs cross-referenced against canonical URLs.

### Common failure modes

- **Framework canonical defaults shipped without audit.** Next.js App Router, headless CMS systems, and locale-aware frameworks ship canonical generation that breaks in subtle ways at scale. Invisible until the site is crawled at scale.
- **Pagination canonicalizing to page 1.** A common pattern in older SEO advice that Google has explicitly walked back.
- **Filter/sort URLs treated inconsistently.** Some filters get canonicalized; some don't; some are blocked in robots.txt; some are noindexed. Without a documented strategy, every team makes a different decision.
- **Canonical decisions made by developers without SEO input.** The canonical tag is a one-line piece of HTML, but the decision behind it is strategic.

### Cross-lever interactions

- **Lever 1 (Crawl Architecture):** Almost always linked. Canonical bugs feed crawl waste. The standard pattern: fix canonicals first, validate crawl recovery.
- **Lever 3 (Page Technical Signals):** Lever 3 implements Lever 2's architecture decisions per page. If Lever 2 is wrong, Lever 3 fixes are rearranging deck chairs.
- **Lever 6 (Programmatic SEO):** Programmatic surfaces multiply canonical bugs if templates ship without canonical strategy validation.
- **Lever 8 (Authority/Trust/GEO):** Canonical confusion fragments authority signals across multiple URLs.

### Default severity weights

- Canonical pointing to a non-existent or 404 URL → **P0**
- Double-locale prefix bug on commercial pages → **P0**
- Canonical pointing to a different URL than internal links target → **P1**
- Pagination canonicalizing to page 1 → **P2**
- Filter URLs without documented canonical strategy → **P2**

## Lever 3 — Page Technical Signals

### What this lever covers

The page-level technical signals the crawler evaluates each time it reaches a URL. Whether the page itself sends the right signals to be indexed, ranked, and understood — separate from the site-level decisions in Levers 1 and 2.

### Why it sits at position 3

This lever implements the architecture decisions from Lever 2 at the page level. Lever 3 is also where most of the work covered by traditional "technical SEO audits" actually lives. It sits at position 3 because if Levers 1 and 2 are broken, fixing Lever 3 signals is rearranging deck chairs.

### What success looks like

Every page returns the right HTTP status code. Renders consistently in raw HTML and rendered DOM. Loads quickly enough that its quality is evaluable. Declares its language correctly. Marks up its entities and content type with valid schema. Is mobile-friendly without a separate mobile experience. Indexability directives are deliberate and audited.

### Diagnostic signals to evaluate

- **HTTP status code distribution** (2xx/3xx/4xx/5xx ratios across the site).
- **Redirect chain depth.** Chains of 3+ hops on commercial URLs are common and costly.
- **307 vs. 301 patterns.** 307 is temporary; should be 301 for permanent moves. Cloudflare and edge platforms sometimes serve 307 by default.
- **hreflang annotations.** Reciprocal alignment, x-default presence, alignment with canonicals.
- **Schema and structured data.** JSON-LD validity, completeness, type coverage. Use Schema Detector skill for single-URL deep inspection.
- **Core Web Vitals.** LCP, INP, CLS. **Field data (CrUX) is the ranking signal.** Lab data (Lighthouse) is diagnostic only.
- **JavaScript rendering parity.** Rendered DOM matches raw HTML for critical content (titles, schema, meta, internal links). Single most common Lever 3 issue.
- **Mobile rendering.** Mobile-first index has been the default since 2019. Audit the mobile rendered DOM, not the desktop one.
- **HTML structure.** Heading hierarchy, semantic markup, meta tags.
- **Indexability directives.** noindex, X-Robots-Tag, meta robots — deliberate and audited?

### Data sources required

- **Critical:** Screaming Frog with JS rendering enabled, configured to match Googlebot Mobile, with `Respect noindex` disabled for staging crawls.
- **Critical:** GSC Core Web Vitals report (CrUX field data).
- **Important:** Schema Detector skill output for sample URLs per template.
- **Important:** GSC URL Inspection for any URL flagged with rendering or indexation issues.
- **Useful:** PageSpeed Insights for individual URLs (provides field + lab data side by side).

### Common failure modes

- **JS rendering parity gaps go undetected.** Critical content added to the DOM after the initial HTML response. Default crawls miss this. Only crawls with JavaScript rendering enabled and properly configured catch the gap. **Most common Lever 3 issue.**
- **Core Web Vitals treated as a Lighthouse score instead of a field-data signal.** Lab data and field data often disagree.
- **Schema implemented inconsistently across templates.** Article schema on some posts, none on others. Inconsistent implementation is treated as no implementation by validators.
- **Redirect chains accumulate over years.** A 2018 URL → 2020 URL → 2023 URL. Each hop costs crawl efficiency and dilutes link equity.

### Cross-lever interactions

- **Lever 2 (Canonical):** Page-level canonical implementation lives here.
- **Lever 8 (Authority/Trust/GEO):** Schema is split between Lever 3 (validity, completeness) and Lever 8 (authority signals like Person `sameAs`). Schema gaps on author pages are both Lever 3 and Lever 8 findings — name both.
- **Lever 5 (Content):** Content rendering and structured data presentation overlap.

### Default severity weights

- Critical content (titles, H1, schema) missing in raw HTML and only added by JS → **P0**
- 5xx errors on commercial URLs → **P0**
- Core Web Vitals failing CrUX thresholds on commercial templates → **P1**
- Redirect chain >3 hops on commercial URLs → **P1**
- Schema missing on templates where it would surface rich results → **P1**
- Suboptimal heading hierarchy on a content cluster → **P3**

## Lever 4 — Keyword Strategy & Opportunity Modeling

### What this lever covers

Identifying which keywords matter, why they matter, and how their pursuit should be sequenced. Translates ICP clarity into a keyword universe, scores that universe by commercial value and competitive feasibility, and produces a roadmap that other levers execute against.

### Why it sits at position 4

By this point, the technical foundation (Levers 1–3) is solid enough that ranking signals can land. Now the question becomes: rank for what? Lever 4 produces the roadmap that drives Lever 5 (content) and Lever 6 (programmatic). Without Lever 4, content production is uncoordinated.

**ICP clarity is a prerequisite.** Where ICP isn't documented, surfacing that gap is the first finding. The work to develop ICP itself is brand strategy, out of Ygramul scope.

### What success looks like

The keyword universe is documented and segmented by intent (informational, commercial, transactional, navigational, brand) and by funnel stage. Every keyword carries a commercial value score and a feasibility score. An explicit opportunity ranking exists. A quarter-by-quarter roadmap is produced and revisited monthly.

### Diagnostic signals to evaluate

- **Keyword universe coverage.** Categories of intent represented and missing.
- **Keyword opportunity scoring.** Commercial value × feasibility. Is there scoring at all, or just volume sorting?
- **Search volume trends and seasonality.** 12-month and 36-month patterns.
- **SERP feature presence per query.** AI Overviews, featured snippets, video, local pack. Determines what "ranking" actually looks like for each query.
- **Competitive density.** How many sites are credibly targeting each query?
- **Keyword cannibalization.** Multiple URLs targeting the same query — sample by template.
- **Branded vs. non-branded query share.** Branded share is a leading indicator for Lever 10.

### Data sources required

- **Critical:** GSC Performance report (Queries × Pages, 16 months).
- **Critical:** SEMrush or Ahrefs keyword universe export for the client's seed keywords + competitors.
- **Important:** SERP feature data per query (Ahrefs, SEMrush, or DataForSEO).
- **Important:** Documented ICP from the client. If absent, declare the gap.
- **Useful:** Google Trends for seasonality validation.

### Common failure modes

- **Keyword universe scoped only to high-volume head terms.** The body and long tail — where commercial activity often lives, and where AI-search query patterns are emerging — gets ignored.
- **Search volume treated as commercial value.** A 50K-volume informational query is not more valuable than a 500-volume bottom-funnel query.
- **Roadmap built once and never revisited.** AI Overviews appear and disappear. A quarterly-static roadmap is stale within six weeks.
- **Cannibalization goes uncaught.** Three URLs target the same query. None ranks well. Pattern was never identified at the keyword strategy level.

### Cross-lever interactions

- **Lever 5 (Content):** Lever 4 produces the brief queue that Lever 5 executes against.
- **Lever 6 (Programmatic):** Programmatic patterns are identified in Lever 4 and built in Lever 6.
- **Lever 10 (Brand Demand):** Branded search trend is a leading indicator for Lever 10 work.
- **Lever 11 (Auto-Research):** Outcome data from Lever 5 and Lever 6 feeds back to update Lever 4 priors.

### Default severity weights

- ICP undocumented or ambiguous → **P0** (blocks all of Lever 4)
- Cannibalization across commercial templates → **P1**
- No SERP feature accounting in roadmap → **P1**
- Volume-only scoring (no intent or feasibility weighting) → **P2**
- Roadmap older than 90 days → **P2**

## Lever 5 — Content Production & Optimization

### What this lever covers

Producing content against the Lever 4 roadmap and maintaining that content as it ages. Briefs, drafting standards, on-page optimization, content decay analysis, message-match issues, and conversion-path friction signals.

### Why it sits at position 5

Content is what Levers 1–4 enable to rank. The technical foundation makes it findable. The canonical architecture makes it indexable. The keyword strategy makes it deliberate. Without those upstream levers solved, content work doesn't compound.

### What success looks like

Content shipped against the roadmap ranks for its target queries. Holds rank against incumbents. Refreshes on a cadence appropriate to its content type. Briefs are detailed enough that drafting produces consistent quality regardless of who writes. Content decay is monitored — pages losing position get caught and refreshed before they drop out of the top 10. Where pages succeed at attracting traffic but fail at converting it, that gap is surfaced.

### Diagnostic signals to evaluate

- **Content brief depth and ranking outcome by brief.** Sample 5–10 briefs and look at the resulting page's ranking.
- **Content decay velocity.** Rate of position loss over time per URL. Look at 30/90/180-day decay curves.
- **On-page optimization completeness.** Title, meta, H-tags, internal links, schema. Modern check; not just "title and meta."
- **Topical depth vs. thin-content patterns.** Word count alone is not the signal; depth of entity coverage relative to top-ranking pages is.
- **Message-match between query intent and page promise.** Does the page deliver on what the query implied?
- **Conversion-path friction.** High-traffic, low-engagement pages.
- **Content freshness signals.** lastModified, dateModified, real content updates (not just timestamp bumps).
- **AIO opener block presence and quality.** Use AIO Cascade skill output as the standard.

### Data sources required

- **Critical:** GSC Performance report segmented by URL.
- **Critical:** Sample of recent briefs (5–10) + their resulting published pages.
- **Important:** Engagement metrics (GA4 or equivalent) for high-traffic URLs.
- **Important:** AIO Cascade skill applied to top-traffic content URLs.
- **Useful:** SERP scrape of the top-3 results for the page's target query, for entity coverage comparison.

### Common failure modes

- **Briefs too thin to produce consistent output.** Writers fill in gaps from intuition. Quality varies wildly by writer.
- **Content decay treated as a content problem when it's often a SERP feature problem.** A page lost rank because Google added an AI Overview. Refreshing the page doesn't fix this. Understanding the new SERP shape does.
- **Optimization stops at the title tag.** Modern on-page work includes schema, internal linking integration, content section structure, entity coverage, and AI-search opener blocks. Check-the-title-and-meta is incomplete.
- **Conversion friction ignored or treated as CRO in isolation.** A page that ranks but doesn't convert is sometimes a content problem — the page promised something the user didn't get. Surfacing this gap is Lever 5; closing it is CRO and out of Ygramul scope.

### Cross-lever interactions

- **Lever 4 (Keyword Strategy):** Briefs come from the Lever 4 roadmap.
- **Lever 7 (Internal Linking):** Internal linking is integrated into the brief, not treated as a separate post-publication step.
- **Lever 8 (Authority/Trust/GEO):** AIO opener blocks and entity-rich content drive AI-search citation.
- **Lever 11 (Auto-Research):** Brief outcome data is the cleanest learning signal in Ygramul.

### Default severity weights

- Briefs missing on commercial templates → **P0**
- Top-10 commercial pages decaying without refresh → **P1**
- AIO opener block missing on top-traffic content URLs → **P1**
- Internal linking absent from brief template → **P2**
- Schema integration missing from on-page work → **P2**

## Lever 6 — Programmatic SEO

### What this lever covers

Identifying repeatable content patterns that warrant production at scale, building templates that produce quality output at that scale, and managing the technical and editorial integrity of the resulting page surface. Location pages, comparison pages, glossary pages, integration pages, any other template-based production where the value is in coverage across instances.

### Why it sits at position 6

Programmatic SEO depends heavily on upstream levers. The canonical architecture (Lever 2) must handle template-driven URL patterns. The keyword strategy (Lever 4) must identify the patterns worth productizing. The content standards (Lever 5) must be encoded into the template. Run programmatically without those dependencies solved, and a single bad template multiplies quality issues across thousands of pages.

### What success looks like

Programmatic page surfaces grow the site's keyword coverage in commercially valuable ways without degrading the site's overall quality signal. Each programmatic page is differentiated enough to avoid being treated as duplicate content. Templates produce content that's genuinely useful, not just keyword-stuffed scaffolding. The programmatic surface integrates cleanly with the rest of the site's information architecture (Lever 7) and respects the canonical strategy (Lever 2).

### Diagnostic signals to evaluate

- **Template pattern identification.** Which patterns warrant programmatic production?
- **Per-instance differentiation.** Does each generated page have unique value?
- **Programmatic content coverage vs. query-pattern coverage gaps.**
- **Quality-signal monitoring.** Does the programmatic surface degrade the site's overall ranking power?
- **Template performance drift.** Do early instances rank? Do later instances rank? What changed?
- **Canonical and indexation behavior on programmatic surfaces.**
- **Template share of total indexed URL count.** If programmatic templates are >70% of indexed URLs, the site's quality signal is dominated by template output.

### Data sources required

- **Critical:** Screaming Frog crawl segmented by template/URL pattern.
- **Critical:** GSC Performance report filtered to programmatic URL patterns.
- **Important:** Sample of 10 instances from each template, manually inspected for differentiation.
- **Important:** Indexation rate per template.
- **Useful:** Time-series ranking data per template (early vs. recent instances).

### Common failure modes

- **Templates produce thin content at scale.** A "best [X] in [Y]" template that's 40% boilerplate, 30% repurposed product data, 30% AI-generated filler ranks for nothing.
- **Programmatic surface bypasses Lever 2.** Templates ship with default canonicals pointing at themselves, ignoring the canonical strategy. Faceted programmatic pages create thousands of near-duplicate URLs.
- **Quality drift goes unmonitored.** Template instances 1–500 rank well; instances 501–5,000 don't. Without per-instance ranking monitoring, the drift is invisible until total traffic drops.
- **Programmatic without content-quality investment.** The template + data = content at scale insight breaks if content quality isn't built into the template.

### Cross-lever interactions

- **Lever 2 (Canonical):** Programmatic + canonical bugs is the highest-multiplier failure mode in Ygramul.
- **Lever 1 (Crawl):** Programmatic surfaces compound crawl waste if Lever 1 isn't solid.
- **Lever 5 (Content):** Template quality is upstream content work.
- **Lever 7 (Internal Linking):** Programmatic pages are often orphaned without explicit Lever 7 work.

### Default severity weights

- Programmatic surface with thin-content template at >1K instances → **P0**
- Programmatic templates without canonical strategy validation → **P0**
- Programmatic surface >70% of indexed URLs without quality monitoring → **P1**
- Template performance drift uncaught → **P1**
- Programmatic pages orphaned in internal link graph → **P2**

## Lever 7 — Internal Link Architecture

### What this lever covers

Designing how the site's pages link to each other — both for crawler distribution and for the flow of authority through the site graph. Internal link discoverability, anchor text patterns, hub-and-spoke topology, and orphan page resolution.

### Why it sits at position 7

By this point, the content asset library exists. The keyword strategy (Lever 4) has identified the destinations that matter. The content production (Lever 5) and programmatic work (Lever 6) have produced the pages. Internal linking is the layer that connects all of this into a coherent site graph. It can only operate on what exists.

### What success looks like

Every commercially valuable page is reachable from the homepage in three or fewer clicks. Every page links to topically related peers. Those links use anchor text that describes the destination accurately. Hub pages aggregate authority and distribute it to the cluster pages they govern. Orphan pages don't exist on commercially valuable URL patterns. The internal link graph is intentional, not accidental.

### Diagnostic signals to evaluate

- **Click depth from homepage per URL.**
- **Orphan page detection by template.** Pages with zero inbound internal links.
- **Anchor text distribution and diversity by destination URL.**
- **Hub-and-spoke topology consistency.** Do hub pages actually link to all their spokes?
- **Inbound internal link count per URL** (proxy for site-internal authority).
- **Cross-template linking patterns.** Does the blog link to product pages? Does documentation link to feature pages?
- **Anchor text relevance to destination content.**

### Data sources required

- **Critical:** Screaming Frog crawl with internal link export (inlinks per URL, anchor text per inlink).
- **Important:** Sitemap cross-referenced against crawled URLs to detect orphans (pages in sitemap but no inbound links).
- **Useful:** GA4 page views by URL (low-traffic URLs that should be high-traffic are orphan candidates).

### Common failure modes

- **Internal linking treated as a navigation problem instead of an authority distribution problem.** Header links to the same five pages from every URL. The body of every page links to nothing else internal. Authority concentrates in navigation targets.
- **Anchor text generic and uninformative.** "Click here," "learn more," "read our blog" — anchor text that doesn't describe the destination tells the crawler nothing about topical relevance.
- **Orphan pages on programmatic surfaces.** Templates ship without integration into the site graph. Pages might be in sitemap but isolated.
- **Hub pages exist but don't link to all their cluster pages.** Hub links to the most prominent six and ignores the next forty. Long tail never benefits from hub authority.

### Cross-lever interactions

- **Lever 5 (Content):** Internal linking should be part of the content brief.
- **Lever 6 (Programmatic):** Programmatic pages are most commonly orphaned.
- **Lever 8 (Authority):** Internal authority distribution amplifies external authority signals.

### Default severity weights

- Commercial pages with zero inbound internal links (orphans) → **P1**
- Click depth >4 from homepage on commercial URLs → **P1**
- Hub pages missing >50% of their cluster spokes → **P2**
- Generic anchor text dominant across the site → **P2**

## Lever 8 — Authority, Trust & GEO Visibility

### What this lever covers

Building the site's recognition by external systems — both classical search authority signals and AI-search citation behavior. Entity infrastructure, organizational legitimacy markers, author credibility (E-E-A-T), inbound link quality, and visibility in AI-search surfaces (Google AI Overviews, Perplexity, ChatGPT search, Gemini).

### Why it sits at position 8

Authority is downstream of content. You can't be cited as authoritative on a topic if you don't have content on that topic. By position 8, the site has the technical foundation, canonical strategy, keyword roadmap, content asset library, and internal link graph. All of those are necessary substrates. Authority work that precedes this substrate is unanchored — citations point at thin pages and ranking gains don't compound.

### What success looks like

The site is recognized as the authoritative source for its topic area, by both classical search engines and AI-search systems. Author entities are properly defined, with `sameAs` links to external profiles and verifiable credentials surfaced in schema. Organizational legitimacy is established. The site is cited by AI Overviews and large language models for queries in its topic area, not just blue-link-ranked. Inbound links are earned from credible sources.

### Diagnostic signals to evaluate

- **Author entity infrastructure.** Person schema, `sameAs` links, credential surfacing.
- **Organizational entity infrastructure.** Organization schema, contact info, About page depth.
- **E-E-A-T signals.** Particularly for YMYL content. Experience, expertise, authoritativeness, trustworthiness.
- **NAP consistency** (where applicable) across the web.
- **Inbound link quality and topical relevance.**
- **AI-search citation rate by query.** How often the site is cited by AI Overviews, Perplexity, ChatGPT, Gemini for queries in its topic area.
- **Citation context.** Primary source, supporting reference, or competitor?
- **Brand mention sentiment and frequency in primary sources.**

### Data sources required

- **Critical:** Schema Detector skill output for author pages and Organization schema.
- **Critical:** AI-search citation tracking (Peec AI, Otterly, or manual sampling on primary commercial queries).
- **Important:** Inbound link profile (Ahrefs or SEMrush).
- **Important:** Sample of 5–10 author bios cross-checked against external profiles.
- **Useful:** Brand mention monitoring (Google Alerts, Mention.com, or equivalent).

### Common failure modes

- **E-E-A-T treated as a checkbox instead of as entity infrastructure.** Adding an author byline isn't E-E-A-T. Building author entities — Person schema, verifiable credentials, `sameAs` links to external profiles, individual author pages with topical authority — is.
- **AI-search citation work treated as separate from classical authority work.** It isn't. The same primary-source signals drive both.
- **Inbound link work treated as the whole of authority.** Backlink campaigns, especially from low-quality networks, produce poor signal at best and penalties at worst.
- **Organizational entity infrastructure ignored.** Service businesses neglect About pages, contact pages, NAP consistency, Organization schema.

### Cross-lever interactions

- **Lever 3 (Page Technical Signals):** Schema validity is Lever 3; schema *content* (sameAs, credentials) is Lever 8.
- **Lever 5 (Content):** AIO opener blocks generated in Lever 5 are read by AI-search systems and drive Lever 8 citation rate.
- **Lever 9 (Local SEO):** Organizational entity infrastructure overlaps heavily with Lever 9 NAP consistency.
- **Lever 10 (Brand Demand):** Brand mentions, primary-source citations, and demand-side signals feed both Lever 8 and Lever 10.

### Default severity weights

- Author pages without Person schema on YMYL content → **P0**
- Organization schema missing or invalid → **P1**
- AI-search citation rate <5% on primary commercial queries → **P1**
- NAP inconsistency across primary citation sources → **P1**
- `sameAs` links missing on author entities with strong external profiles → **P2**

## Lever 9 — Local SEO & Google Business Profile

### What this lever covers

Building visibility for place-bound businesses where the customer's search intent includes a geographic component. GBP optimization, local citations, review management, local landing pages, and local schema.

### Why it sits at position 9

Local SEO depends on most upstream levers being solved. The site's technical foundation must be sound — a local business with a broken canonical strategy is invisible regardless of GBP work. The content asset library must include local-relevant content. The authority and entity infrastructure provides the substrate.

**Lever 9 is conditional.** Many Ygramul engagements skip it entirely because the client isn't place-bound (SaaS, B2B, e-commerce with national reach). When a client is place-bound, Lever 9 often delivers the fastest visibility lift in the engagement because the competitive bar in many local categories is low.

### What success looks like

The business appears in the map pack for its primary commercial queries within its service area. The Google Business Profile is complete, current, and actively maintained. Reviews accumulate at a credible cadence and are responded to with care. Citations are accurate and consistent. NAP is identical across every surface. Local landing pages are differentiated, useful, and link cleanly into the rest of the site.

### Diagnostic signals to evaluate

- **GBP completeness.** Services, attributes, hours, photos, posts, products, Q&A.
- **Review velocity, sentiment, and response rate.**
- **Review schema and review surfacing on the site.**
- **Local citation accuracy and consistency** across the citation graph.
- **NAP consistency** between GBP, the site, and external citations.
- **Local schema implementation.** LocalBusiness, Service, GeoCoordinates, OpeningHoursSpecification.
- **Local landing page differentiation.** City pages, service-area pages, neighborhood pages.
- **Map pack and Local Pack ranking signals** for primary commercial queries.
- **Inbound links from locally relevant sources.**

### Data sources required

- **Critical:** Direct access to or audit-mode access to the client's GBP.
- **Critical:** Local rank tracking for primary commercial queries (Local Falcon, BrightLocal, or equivalent).
- **Important:** Citation audit (BrightLocal, Whitespark, or equivalent).
- **Important:** Sample of 5+ local landing pages for differentiation review.
- **Useful:** Review sentiment data over time.

### Common failure modes

- **GBP treated as profile creation rather than ongoing optimization.** Set up once, then sits for two years.
- **Citations built once, never audited.** Phone number changes; site updates; half the citation graph doesn't.
- **Reviews ignored as a ranking signal.**
- **Local landing pages treated as duplicate content.** Five service areas, five pages with the city name swapped.
- **Local schema missing or incomplete.** LocalBusiness schema with no Service entries, no GeoCoordinates, no OpeningHoursSpecification.

### Cross-lever interactions

- **Lever 8 (Authority):** Organization schema and NAP consistency overlap.
- **Lever 5 (Content):** Local landing page content is Lever 5 work scoped to local.
- **Lever 3 (Page Technical Signals):** Local schema is Lever 3 implementation of Lever 9 strategy.

### Default severity weights

- GBP missing or unverified → **P0**
- NAP inconsistency on commercial citations → **P1**
- Local landing pages with thin or duplicate content → **P1**
- Local schema missing or incomplete → **P1**
- Review request workflow absent → **P2**

## Lever 10 — Brand & Search Demand Generation

### What this lever covers

The long-arc work of growing demand for the brand itself — branded search volume, share of voice for category terms, brand mention frequency, and the demand-side keyword universe that feeds future ranking opportunities. Operates on multi-year timescales and runs in parallel to engagement work, not sequenced after it.

### Why it sits in the long-arc grouping

Brand demand generation does not respond to single-engagement work on the timescales engagement levers operate on. Lever 1's crawl waste fixes can show indexation recovery in 30 days. Lever 4's keyword strategy work can show ranking shifts in 90 days. Brand demand growth compounds over 12 to 36 months and depends as much on the quality of all the upstream levers as on dedicated brand work itself.

This is why Lever 10 sits in parallel rather than sequenced last. Effective engagements start brand demand measurement on day one, not at month 12.

### What success looks like

Branded search volume grows over time, faster than the category's overall growth. The brand owns share of voice for category-defining terms even when not directly ranking for those terms (because users search for the brand first). The demand-side keyword universe — questions, problems, jargon used by the ICP — grows the addressable surface for future content work. The brand is cited and mentioned in primary sources, podcasts, conferences, and adjacent industry coverage.

### Diagnostic signals to evaluate

- **Branded search volume trend.** 12 and 36 months.
- **Branded vs. non-branded query share.**
- **Share of voice for category-defining terms.**
- **Brand mention frequency in primary sources.**
- **Demand-side keyword universe expansion.** New questions, new jargon, new problem framings entering the search landscape.
- **Direct and referral traffic share.**
- **Brand authority signals.** Wikipedia presence, knowledge panel completeness, Google entity recognition.

### Data sources required

- **Critical:** GSC Performance report filtered to branded queries (12+ months).
- **Critical:** Google Trends for branded term volume over time.
- **Important:** GA4 for direct/referral traffic share.
- **Important:** Brand mention monitoring tool.
- **Useful:** Knowledge graph entity check (`Organization` entity recognized by Google).

### Common failure modes

- **Branded search treated as a vanity metric instead of a leading indicator.** Branded search volume is one of the strongest leading indicators for organic traffic durability.
- **Brand work treated as out of scope for SEO methodology.** PR, podcast outreach, and conference speaking are not classical SEO disciplines, but their downstream effects on branded search and category authority are measurable in SEO metrics. Surfacing where these effects are happening — and where they aren't — is appropriate Ygramul work even when execution is handled outside.
- **Confusing brand demand with brand awareness.** Brand awareness can grow without producing branded search volume.
- **Neglecting the demand-side keyword universe.** New jargon and problem framings enter the search landscape continuously.

### Cross-lever interactions

- **Lever 4 (Keyword Strategy):** Demand-side keyword universe expansion feeds Lever 4 priors.
- **Lever 8 (Authority):** Brand mentions and primary-source citations overlap with Lever 8 authority signals.
- **Lever 11 (Auto-Research):** Branded search trend is one of the cleanest long-arc signals to track in the auto-research loop.

### Default severity weights

- Branded search trending down over 12 months → **P1** (signals organic traffic durability risk)
- Brand entity not recognized by Google's knowledge graph → **P1**
- Demand-side keyword universe untracked → **P2**
- Brand mention monitoring absent → **P2**

## Lever 11 — Auto-Research & Outcome Measurement

### What this lever covers

The cross-engagement work of measuring what shipped, scoring its outcomes, and feeding the resulting patterns back to update the rules used in Levers 1 through 10. The framework's learning loop. Operates continuously across the full client base.

### Why it sits as the cross-engagement layer

Lever 11 is recursive. Its outputs update the rules used by Levers 1 through 10 in subsequent engagements. This isn't a sequential position. It's a feedback layer that observes everything else and modifies the framework over time.

This is what differentiates a methodology from a checklist. A checklist runs the same way regardless of outcome history. A methodology with auto-research updates its own rules as evidence accumulates.

### Calibration on current state

The auto-research loop is **designed and partially operational**. Outcome measurement on individual findings happens today. The cross-engagement pattern recognition layer — where findings from one engagement inform recommendations on another — is being instrumented, not yet running at full scale.

When applying Lever 11 in audit work, this calibration matters: the loop produces conditional rules in theory; in practice, V1 audits cite default severity weights that are *informed by* but not *yet validated by* cross-engagement outcome data.

### What success looks like

Every recommendation produced by Levers 1–10 is tied to a measurable hypothesis. Outcomes are tracked at consistent cadences (typically 30, 60, and 90 days post-implementation). Findings that consistently predict outcomes get reinforced in the methodology's rules. Findings that consistently fail to predict outcomes get demoted or removed. The version of Ygramul applied to a new engagement is more accurate than the version applied to one a year ago.

### Diagnostic signals to evaluate

- **Per-finding hypothesis tracking.** What was predicted to move?
- **Outcome scoring.** Win, Neutral, Regression — based on impressions, CTR, position, conversion proxies.
- **Pattern recognition across engagements.** Which finding categories predict outcomes most reliably?
- **Rule update cadence.** How often do framework priors shift?
- **Cross-vertical pattern detection.** Do findings that work in SaaS also work in publishing, commerce, local? Where do they not?
- **Calibration drift.** Are the framework's predictions becoming more or less accurate over time?

### Data sources required

For Lever 11 evaluation in an audit context (where applicable), the data sources are operational rather than client-side:

- **Critical:** Internal Ygramul outcome tracking (per-finding hypothesis + 30/60/90 day measurement).
- **Critical:** GSC API access for the client (to measure outcomes post-implementation).
- **Important:** Cross-engagement pattern data (when available — partially built).

For client-facing audit work, Lever 11 typically does not produce findings against the client's site. It produces findings about *the audit's own predictions* and is reflected in the confidence scores attached to other levers' findings.

### Common failure modes

- **Recommendations shipped without measurable hypotheses.** No hypothesis = no scoring = no learning.
- **Outcome measurement on too short a horizon.** A 7-day check on a fix that compounds over 90 days underrates slow-acting work and overrates fast-acting work.
- **Pattern recognition overfit to recency.** Latest 10 outcomes weighted more than the prior 100. A bad month in algorithm noise causes the framework to discard rules that were working.
- **Auto-research treated as a backend process invisible to engagements.** When framework rules update but updates aren't surfaced to engagement work, the learning is lost.

### Cross-lever interactions

Lever 11 interacts with **all** of Levers 1–10. It is the layer that observes their outcomes and updates their priors.

The strongest signal sources are typically:
- **Lever 5 (Content):** Brief outcomes are clean, frequent, and short-cycle.
- **Lever 1 (Crawl):** Indexation recovery is measurable in 30 days.
- **Lever 8 (Authority/GEO):** AI-search citation lift is measurable but slower (60–90 days).

The hardest signal sources:
- **Lever 10 (Brand Demand):** 12–36 month timescales make pattern recognition slow.
- **Lever 6 (Programmatic):** Outcomes can be confounded by template drift over time.

### Default severity weights

Lever 11 doesn't typically generate P0–P3 findings against client sites. When it does (rare), they're meta-findings about the audit's own measurability:

- Client GSC API access blocked → **P0** (blocks outcome measurement entirely)
- Client unwilling to ship fixes with hypothesis tracking → **P1** (degrades auto-research signal quality)
- 30/60/90 measurement cadence not agreed → **P2**

## How to apply this framework in an audit

### Workflow (per Doc 1 master prompt)

1. **Pre-audit intake.** Confirm data sources available. Declare gaps.
2. **Crawl + GSC export.** Run Screaming Frog (with JS rendering, Googlebot Mobile, segmented sitemaps if available). Export GSC Performance + Page Indexing.
3. **Apply Levers 1–9 in order.** Each lever's diagnostic signals → findings → severity. **Sequence matters.** Don't move to Lever 3 if Lever 2 is broken — note the dependency and fix Lever 2 first.
4. **Apply Lever 10 in parallel.** Brand demand signals don't sequence after engagement levers. Pull them at any point in the audit.
5. **Apply Lever 11 reflexively.** For every finding produced in Levers 1–10, write the hypothesis, the predicted metric, the predicted direction, and the measurement cadence.
6. **Cross-lever flagging.** Where a finding spans levers, name both. Sequence the fix by dependency order.
7. **Verification layer** (per Doc 1). Multi-lens critique: at minimum, *technical correctness* + *commercial relevance* + *implementation feasibility*. For Mini/Full audits, add *measurement clarity* + *cross-engagement pattern fit*.
8. **Per-finding status.** 🟢 Validated · 🟡 Hypothesis · 🔴 Disputed · ⚫ Insufficient data.
9. **Deliverable assembly.** Per Doc 5 runbook (when written). Excel workbook + executive narrative + Jira-ready ticket pack.

### What "applying a lever" actually means

For each lever, walk the eight-section structure:

1. Confirm the lever applies to this client (Lever 9 may not).
2. Confirm the data sources required are in hand (declare gaps if not).
3. Evaluate the diagnostic signals against the data.
4. Identify findings — patterns where the signal indicates a problem.
5. Score severity using the default weights, adjusted for client context.
6. Flag cross-lever interactions.
7. Tie each finding to a measurable hypothesis (Lever 11 layer).
8. Apply the verification layer (multi-lens critique + status).
